# Three.js Environment Design

## Purpose

The 3D environment should become the visual and interactive core of the application. It should render saved pile designs, let users inspect and edit design objects, show conflicts and measurements, and accept structured JSON generated by the editor or AI chat.

The design goal is to separate:

- Durable design data stored in the database.
- Derived geometry calculated from design data.
- Three.js render objects used only for visualization.
- UI/editor state such as selection, camera, annotations, and previews.

This separation keeps the application easier to persist, validate, render, version, and extend.

## Design Principles

- Store engineering/design intent, not raw Three.js objects.
- Generate Three.js meshes from canonical data.
- Keep object ids stable across saves and versions.
- Use deterministic geometry functions for calculated objects.
- Treat imported meshes and textures as visual assets, not the source of truth for engineering geometry.
- Store units explicitly.
- Keep visual settings separate from design values.
- Support preview state before committing changes to a saved design.
- Make every object addressable by id so the editor, reports, and AI chat can refer to the same thing.

## Scene Layers

The environment should be organized into layers:

- `design`: real project geometry such as piles, helices, caps, brackets, and soil layers.
- `analysis`: conflicts, clearance zones, load arrows, design checks, and warnings.
- `annotation`: labels, measurements, notes, section cuts, and callouts.
- `environment`: floor, grid, axes, lighting, camera presets, background, and site context.
- `preview`: unsaved proposed changes from the AI chat or editor.
- `debug`: axes helpers, wireframes, bounding boxes, performance overlays.

Layer separation allows users to toggle visibility, export clean reports, and preview AI-generated changes without polluting saved design data.

## Canonical Scene Data

The database should store a canonical design JSON snapshot. The frontend should convert that snapshot into renderable Three.js objects.

Suggested top-level scene structure:

```json
{
  "schemaVersion": "0.1.0",
  "units": {
    "length": "m",
    "angle": "deg",
    "force": "kN"
  },
  "coordinateSystem": {
    "upAxis": "y",
    "horizontalAxes": ["x", "z"],
    "origin": "project"
  },
  "objects": [],
  "relationships": [],
  "materials": [],
  "assets": [],
  "analysis": [],
  "viewState": {}
}
```

Alternative storage shape:

- Store domain-specific arrays such as `piles`, `helixGroups`, `soilLayers`, and `annotations`.
- Generate a unified object graph at runtime.

Recommended early approach:

- Use domain-specific arrays for clarity.
- Add a generic `objects` abstraction later if object types become much broader.

## Object Identity

Every design object should have:

- Stable `id`.
- Human-readable `label`.
- `type`.
- Optional `parentId`.
- Optional `groupId`.
- `createdAt` and `updatedAt`.
- `createdBy` and `updatedBy` for team features.
- `source` indicating manual, imported, generated, or AI-generated.

Example:

```json
{
  "id": "pile-01",
  "type": "pile",
  "label": "Pile 1",
  "source": "manual",
  "transform": {
    "position": { "x": 0, "y": 0, "z": 0 },
    "rotation": { "x": 0, "y": 0, "z": 5 },
    "scale": { "x": 1, "y": 1, "z": 1 }
  }
}
```

The application should never store live Three.js `Mesh`, `Geometry`, or `Material` instances in project data.

## Core Object Types

### Project Environment

Objects:

- `sceneSettings`
- `grid`
- `floorPlane`
- `gradePlane`
- `worldAxes`
- `cameraPreset`
- `lightingPreset`
- `background`

Purpose:

- Control how a design is displayed.
- Support report-friendly camera presets.
- Provide spatial reference.

Persistence:

- Store as view/display state, not engineering data.

### Pile Group

Objects:

- `pileGroup`
- `layoutRule`
- `pile`

Pile group fields:

- `layoutType`: circular, line, rectangular_grid, freeform.
- `spacingRadius`.
- `rowSpacing`.
- `columnSpacing`.
- `count`.
- `locked`.
- `origin`.
- `rotation`.

Pile fields:

- `length`.
- `diameter`.
- `batterAngle`.
- `position`.
- `rotation`.
- `materialId`.
- `status`.
- `overrideGroupLayout`.

Three.js generation:

- Shaft: `CylinderGeometry`.
- Tip: `ConeGeometry`.
- Optional cap/bracket: `BoxGeometry`, `CylinderGeometry`, or imported mesh.
- Use instancing for many identical shafts.

### Helix Group

Objects:

- `helixGroup`
- `helix`

Helix group fields:

- `pileId`.
- `distanceFromBottom`.
- `spacing`.
- `placementRule`.

Helix fields:

- `diameter`.
- `thickness`.
- `rise`.
- `rotations`.
- `segmentsPerStep`.
- `radius`.
- `positionOverride`.
- `materialId`.

Three.js generation:

- Early prototype: deformed `BoxGeometry`, as currently implemented.
- Better production approach: generated parametric ribbon mesh.
- Alternative: `TubeGeometry` for spiral centerline plus plate thickness approximation.
- For performance: cache geometry by helix parameters and reuse material instances.

### Soil And Site Context

Objects:

- `soilLayer`.
- `waterTable`.
- `gradeSurface`.
- `excavation`.
- `frostDepthPlane`.
- `propertyLine`.
- `existingStructure`.

Three.js generation:

- Soil layers: transparent boxes or planes.
- Grade: plane or imported terrain mesh.
- Excavation: extruded shape or clipped volume.
- Water table/frost depth: translucent plane.
- Existing structures: simple boxes initially, imported meshes later.

Persistence:

- Store layer elevations, thicknesses, soil descriptions, colors, and visibility.
- Store imported site meshes as asset references.

### Loads And Design Forces

Objects:

- `loadCase`.
- `pointLoad`.
- `lateralLoad`.
- `moment`.
- `loadCombination`.

Three.js generation:

- Arrows using cylinders/cones or helper objects.
- Labels and magnitude callouts.
- Color-coded load types.

Persistence:

- Store as design inputs separate from visualization.
- Link loads to piles, pile groups, caps, or project origins.

### Analysis Objects

Objects:

- `conflictMarker`.
- `clearanceZone`.
- `measurement`.
- `warningMarker`.
- `passFailRegion`.
- `calculationTrace`.

Three.js generation:

- Clearance zones: translucent cylinders/spheres/disks.
- Conflicts: red line segments between closest points, highlighted helices, warning labels.
- Measurements: line geometry plus label sprite/html overlay.

Persistence:

- Store calculation results with design version and calculation engine version.
- Do not treat analysis output as source geometry.

### Annotation Objects

Objects:

- `label`.
- `note`.
- `dimension`.
- `sectionCut`.
- `cameraBookmark`.
- `reportView`.

Three.js generation:

- HTML overlays for labels.
- Line segments for dimensions.
- Plane helpers for section cuts.

Persistence:

- Store in design snapshot if they should travel with the design.
- Store in report-specific data if they only support an export.

### Asset Objects

Objects:

- `meshAsset`.
- `textureAsset`.
- `materialPreset`.
- `symbolAsset`.
- `logoAsset`.

Use cases:

- Manufacturer-specific pile caps or brackets.
- Site context meshes.
- Realistic material textures.
- Report branding.

Persistence:

- Store asset metadata in the database.
- Store actual binary files in object storage.
- Reference assets by id and version in design snapshots.

## Ways To Represent Objects In Three.js

### Primitives

Best for:

- Engineering geometry.
- Parametric objects.
- Fast editing.
- Accurate dimensions.
- Stable exports.

Examples:

- Pile shaft: `CylinderGeometry`.
- Pile tip: `ConeGeometry`.
- Pile cap: `BoxGeometry` or `CylinderGeometry`.
- Soil layer: `BoxGeometry`.
- Grade/floor: `PlaneGeometry`.
- Load arrow: cylinder plus cone.
- Clearance zone: `SphereGeometry`, `CylinderGeometry`, or `TorusGeometry`.
- Measurement line: `BufferGeometry` line segments.

Advantages:

- Lightweight.
- Easy to regenerate from data.
- Easy to validate dimensions.
- No asset pipeline required.

Limitations:

- Less visually realistic.
- Complex components can look generic.

### Parametric And Custom Geometry

Best for:

- Helices.
- Spiral plates.
- Custom pile profiles.
- Clearance envelopes.
- Terrain simplifications.

Approaches:

- Generate `BufferGeometry` directly.
- Use `ExtrudeGeometry` for profiles.
- Use `LatheGeometry` for rotational parts.
- Use `TubeGeometry` for curves.
- Generate ribbons/surfaces from curves.

Advantages:

- Accurate enough for visual engineering concepts.
- Fully controlled by data.
- Can be cached by parameter hash.

Limitations:

- More code to maintain.
- Needs tests to ensure geometry is correct.
- Can be expensive if regenerated too often.

### Instanced Meshes

Best for:

- Many piles.
- Repeated helices with identical geometry.
- Repeated markers or labels.

Advantages:

- Better performance.
- Lower draw calls.

Limitations:

- Harder per-object material variation.
- Harder selection/picking unless ids are carefully managed.

### Imported Meshes

Best for:

- Manufacturer-specific components.
- Detailed brackets, caps, equipment, and site context.
- Client-facing presentations.

Preferred format:

- glTF/GLB.

Other possible formats:

- OBJ for simple meshes.
- STL for engineering/manufacturing assets.
- FBX only if needed.

Advantages:

- High visual fidelity.
- Can represent objects too complex for primitives.
- Good for presentation and manufacturer libraries.

Limitations:

- Harder to validate engineering dimensions.
- Requires asset management.
- May be large.
- Can create licensing/IP issues.
- Needs LOD and optimization.

Recommended rule:

- Use canonical data and primitives/parametric geometry for engineering objects.
- Use imported meshes for optional visual skins or non-critical context.

### Materials And Textures

Material types:

- Basic material for debug.
- Standard/physical material for realistic rendering.
- PBR texture sets for galvanized steel, concrete, soil, water, and painted steel.
- Transparent materials for soil layers and clearance zones.
- Emissive or colored materials for warnings.

Texture maps:

- Base color.
- Normal.
- Roughness.
- Metalness.
- Ambient occlusion.
- Height/displacement where useful.

Material persistence:

- Store material presets by id.
- Store visual overrides separately from domain geometry.
- Keep analysis colors consistent and theme-controlled.

## Editor State Vs Saved State

Saved design state:

- Units.
- Piles.
- Helices.
- Soil/site inputs.
- Loads.
- Design settings.
- Materials/assets references.
- Annotations intended to persist.

Transient editor state:

- Current selected object.
- Hovered object.
- Open panel.
- Unsaved draft values.
- Camera orbit position unless explicitly saved.
- Temporary AI preview.
- Drag handles.
- Debug overlays.

Report state:

- Camera bookmark.
- Visible layers.
- Snapshot image id.
- Report annotations.
- Export settings.

## Preview And AI Generated Objects

The environment should support proposed changes without committing them.

Preview object fields:

- `previewId`.
- `sourceRunId`.
- `baseDesignVersionId`.
- `proposedPatch`.
- `expiresAt`.
- `status`: pending, accepted, rejected, superseded.

UI behavior:

- Render preview objects with distinct material treatment.
- Allow side-by-side or overlay comparison.
- Show a patch summary.
- Require user approval before saving.

## Selection And Picking

Each rendered Three.js object should map back to a design object id.

Needed features:

- Click pile to select pile.
- Click helix to select helix.
- Hover highlight.
- Multi-select for conflict checks.
- Select from table and highlight in scene.
- Select conflict marker and highlight related helices.

Implementation idea:

- Store `object.userData.designObjectId`.
- Store `object.userData.type`.
- Use raycasting for picking.
- Keep selection state as ids, not object references.

## Derived Geometry Pipeline

Suggested runtime pipeline:

1. Load design snapshot.
2. Validate schema.
3. Convert units to render units.
4. Derive pile positions from layout rules.
5. Derive helix positions from pile and helix group rules.
6. Generate or retrieve cached geometries.
7. Apply materials.
8. Register pickable objects.
9. Generate analysis overlays.
10. Render scene.

The same derivation functions should be testable outside React and Three.js.

## Versioning And Persistence

Design snapshots should be immutable once saved as a version.

Store:

- Canonical design JSON.
- Calculation result JSON.
- Thumbnail/render preview.
- Schema version.
- Created by.
- Created at.
- Parent version id.

Use patches for collaboration and autosave, but store full snapshots for stable version history.

## Performance Considerations

Likely bottlenecks:

- High pile and helix counts.
- High segment counts on helices.
- Frequent geometry regeneration.
- Large imported meshes.
- Texture memory.
- HTML labels over many objects.

Mitigations:

- Cache generated geometry by parameter hash.
- Use instanced meshes where practical.
- Debounce expensive updates while dragging.
- Render low-detail previews during interaction.
- Use LOD for imported meshes.
- Compress textures.
- Lazy-load asset libraries.
- Keep analysis overlays toggleable.

Possible future features:

- 2D plan view synchronized with 3D view.
- Section cuts.
- Measurement tools.
- Drag-to-position piles.
- Parametric templates.
- Soil layer editor.
- Load visualization.

## Key Decisions

- What is the canonical design schema?
- Which object types are required for the paid MVP?
- Which units are supported at launch?
- Which geometry is engineering-critical and must remain parametric?
- Which visual assets can be imported meshes?
- Should material presets be global, organization-specific, or project-specific?
- How should AI previews be shown and accepted?
- What maximum scene complexity should be supported smoothly?
- Which 3D interactions are required before adding project persistence?
